## 2. 文档目的与适用范围

### 2.1 目的

定义基于 **AI Agent（Claude Code / Open Code）+ Playwright CLI** 的端到端（E2E）测试标准，规范测试用例编写、执行流程和结果报告，实现：

- 用自然语言编写测试用例，无需编写代码
- AI 自动理解语义并执行测试
- 统一的测试用例格式和报告模板

### 2.2 适用范围

本标准适用于 Web 应用的以下测试场景：

| 场景 | 是否适用 |
|------|---------|
| 功能回归测试 | ✅ |
| 用户路径验证 | ✅ |
| 界面交互测试 | ✅ |
| 性能测试 | ❌ |
| 单元测试 | ❌（使用 Vitest/Jest 等） |
| API 测试 | ❌（使用 API 测试工具） |

## 3. 测试方法概述

### 3.1 核心原理

```
┌─────────────────────────────────────────────────────────────────┐
│  测试用例（自然语言）                                             │
│  e2e/test-cases/{feature}.md                                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  AI Agent (Claude Code / Open Code)                             │
│  - 解析测试用例语义                                              │
│  - 生成 playwright-cli 命令                                     │
│  - 执行并验证结果                                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Playwright CLI                                                 │
│  - 控制浏览器执行操作                                            │
│  - 截图、快照、网络监控                                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  测试报告                                                        │
│  e2e/reports/TEST-REPORT-YYYYMMDD-HHMMSS.md                    │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 模块划分原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 按功能分 | 每个业务功能一个目录 | auth/、user/、order/ |
| 按层级分 | 按页面或组件层级划分 | user/、user/profile/、user/settings/ |

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 功能模块 | 小写英文，复数形式 | auth/、users/、orders/、products/ |
| 单一实体 | 小写英文，单数形式 | profile/、settings/、dashboard/ |
| 测试类型 | 描述性名称 | smoke/、integration/、regression/ |

## 5. 测试用例编写规范

### 5.1 文件与用例的关系

**一个测试文件包含多个相关的测试用例**，按功能/操作分类组织。

```
user-edit.md (文件)
├── 用例1: 正常编辑用户
├── 用例2: 编辑不存在的用户
├── 用例3: 编辑时字段验证
├── 用例4: 编辑后取消操作
└── 用例5: 编辑时并发冲突
```

**组织原则**：

| 原则 | 说明 | 示例 |
|------|------|------|
| 按功能分 | 同一功能的不同场景放在一个文件 | `login.md` 包含正常登录、错误密码、空用户名等 |
| 按操作分 | 同一操作的各种测试放在一个文件 | `user-edit.md` 包含编辑的各种情况 |
| 正常+异常 | 覆盖正常路径和异常路径 | 成功、失败、边界条件 |
| 独立性 | 每个用例可单独执行，互不依赖 | 用例之间不共享状态 |

**文件拆分规则**：

当某个功能的测试用例**过多**时（建议超过 10 个用例），可以按以下方式拆分：

```
# 示例1：按复杂度拆分
user/
├── user-edit-basic.md      # 基础编辑功能（3-5个用例）
├── user-edit-validation.md # 字段验证相关（3-5个用例）
└── user-edit-permission.md # 权限相关（2-3个用例）

# 示例2：按测试类型拆分
order/
├── order-create.md         # 创建订单相关
├── order-payment.md        # 支付相关
└── order-shipping.md       # 物流相关
```

### 5.2 文件命名规范

| 类型 | 命名规则 | 示例 | 包含内容 |
|------|---------|------|---------|
| 功能操作 | `{feature}-{action}.md` | `login.md`, `user-edit.md` | 该操作的多种测试场景 |
| 测试套件 | `suite-{name}.md` | `suite-auth.md` | 跨模块的测试集合 |
| 冒烟测试 | `smoke-test.md` | `smoke-test.md` | 核心功能的快速验证 |

### 5.3 测试用例结构

每个测试用例文件应包含以下部分：

```markdown
# {用例标题}

**优先级**: P0 / P1 / P2 / P3
**模块**: {功能模块}
**前置条件**: {执行前需要满足的条件}
**预计耗时**: {秒数}

## 用例N: {用例名称}

### 步骤
1. [操作描述]
2. [操作描述]
3. ...

### 验证
- [验证点1]
- [验证点2]

### 清理
- [清理操作，如删除测试数据]
```

### 5.4 操作步骤编写规范

#### 支持的操作类型

| 操作类型 | 语法示例 | 说明 |
|---------|---------|------|
| 导航 | `导航到 {URL}` | 打开指定 URL |
| 点击 | `点击 {元素描述}` | 点击元素 |
| 输入 | `输入 {文本}` | 在当前焦点输入 |
| 填写 | `填写 {字段名}={值}` | 填写指定字段 |
| 选择 | `选择 {选项}` | 选择下拉选项 |
| 等待 | `等待 {N} 秒` | 等待指定时间 |
| 截图 | `截图 [{文件名}]` | 保存截图 |
| 验证 URL | `验证 URL 包含 {路径}` | 检查当前 URL |
| 验证元素 | `验证存在 {元素描述}` | 检查元素存在 |
| 验证文本 | `验证包含 {文本}` | 检查页面文本 |
| 滚动 | `滚动到 {元素描述}` | 滚动到指定位置 |

#### 编写原则

1. **使用自然语言**：描述清晰，避免歧义
2. **明确验证点**：每个用例必须有明确的验证标准
3. **可重复执行**：用例之间相互独立，可重复执行
4. **清理测试数据**：执行后恢复初始状态

### 5.5 完整示例

**单文件多测试用例示例 - `user-edit.md`**：


## 6. 测试执行

> - `playwright-cli --help` - 本地命令帮助

### 6.1 使用 AI Agent 执行

#### Claude Code 执行示例

```
使用 playwright-cli 执行 e2e/test-cases/auth/login.md 中的所有测试用例：
- 使用会话名称 test-session
- 有头模式（--headed）
- 每个用例执行后截图
- 生成测试报告到 e2e/reports/
```

#### Open Code 执行示例

```
请执行 e2e/test-cases/user/crud.md 中的测试用例
要求：
1. 每个步骤执行前截图
2. 失败时保留浏览器状态
3. 报告包含执行时间
```

### 6.2 会话管理

```bash
# 创建独立测试会话
playwright-cli -s=test-session open {应用URL} --persistent

# 查看运行中的会话
playwright-cli list

# 打开监控面板
playwright-cli show

# 测试结束后清理
playwright-cli -s=test-session close
```

### 6.3 批量执行

创建 `e2e/run-all-tests.md`:

```markdown
# 批量执行所有测试

## 执行顺序
1. e2e/test-cases/auth/login.md
2. e2e/test-cases/user/crud.md
3. e2e/test-cases/smoke/smoke-test.md

## 全局配置
- 会话名：test-session
- 有头模式：是
- 失败重试：1次
- 报告目录：e2e/reports/
```

### 6.4 测试进度管理

当测试用例较多或执行时间较长时，支持保存测试进度并断点续传。

#### 进度文件格式

**e2e/progress/test-progress.txt**:

```
# 格式：{编号} {状态} {时间}
# 状态：PASS/FAIL/SKIP/空(未执行)

auth_login_normal_1 PASS 2026-03-16_14:30:15
auth_login_normal_2 PASS 2026-03-16_14:30:45
auth_login_err_pwd_1 PASS 2026-03-16_14:31:20
auth_login_err_empty_1 FAIL 2026-03-16_14:31:55
auth_logout_normal_1 PASS 2026-03-16_14:32:10

user_create_normal_1 PASS 2026-03-16_14:32:50
user_create_normal_2
user_create_err_duplicate_1
user_edit_basic_1
user_edit_validation_1
```

**说明**：
- 一行一个测试用例
- PASS = 已通过
- FAIL = 失败
- 空 = 未执行
- 每行包含：编号、状态、执行时间

#### 测试编号规范

```
{模块}_{操作}_{场景}_{类型}_{序号}
```

| 部分 | 说明 | 可选值 | 示例 |
|------|------|--------|------|
| 模块 | 功能模块 | auth, user, order, data | `user` |
| 操作 | 具体操作 | login, create, edit, delete, list | `edit` |
| 场景 | 测试场景 | normal, err, boundary | `err` |
| 类型 | 具体类型 | pwd, empty, notfound, duplicate | `notfound` |
| 序号 | 同类序号 | 1-9 | `1` |

**编号示例**：

| 编号 | 说明 |
|------|------|
| `auth_login_normal_1` | 认证-登录-正常流程-第1个 |
| `auth_login_err_pwd_1` | 认证-登录-密码错误-第1个 |
| `auth_login_err_empty_1` | 认证-登录-空用户名-第1个 |
| `user_create_normal_1` | 用户-创建-正常流程-第1个 |
| `user_create_err_duplicate_1` | 用户-创建-重复错误-第1个 |
| `user_edit_basic_1` | 用户-编辑-基础功能-第1个 |
| `user_edit_validation_1` | 用户-编辑-字段验证-第1个 |
| `order_payment_normal_1` | 订单-支付-正常流程-第1个 |
| `data_query_perf_1` | 数据-查询-性能测试-第1个 |

#### AI 自动生成进度文件

**首次执行**：当进度文件不存在时，AI 自动扫描所有测试用例并生成进度文件。

给 AI Agent 的提示：

```
执行 e2e/test-cases/ 目录下的所有测试用例：

1. 检查 e2e/progress/test-progress.txt 是否存在
2. 如果不存在：
   - 扫描所有测试用例文件
   - 提取每个用例的编号
   - 生成初始进度文件（所有状态为空）
   - 保存到 e2e/progress/test-progress.txt
3. 如果存在：
   - 读取进度文件
   - 跳过状态为 PASS 的测试
   - 从第一个空状态的测试开始执行
4. 每个测试执行后更新对应行的状态和时间
```

#### 进度管理命令

```bash
# 重置进度（清空状态）
sed -i 's/ PASS .*$//' e2e/progress/test-progress.txt
sed -i 's/ FAIL .*$//' e2e/progress/test-progress.txt

# 重试失败的测试
sed -i 's/FAIL.*$/ /' e2e/progress/test-progress.txt

# 完全重置（重新生成）
rm e2e/progress/test-progress.txt
```

## 8. 最佳实践

### 8.1 测试用例设计

| 原则 | 说明 |
|------|------|
| 独立性 | 用例之间不依赖，可单独执行 |
| 可重复 | 多次执行结果一致 |
| 可维护 | 用例结构清晰，易于修改 |
| 覆盖全面 | 正常路径 + 异常路径 |
| 优先级明确 | P0(核心) > P1(重要) > P2(一般) > P3(低) |

### 8.2 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 元素找不到 | 页面未加载完成 | 增加等待时间 |
| 操作失败 | 选择器变更 | 使用更稳定的定位方式 |
| 测试不稳定 | 网络延迟 | 增加重试机制 |
| 数据污染 | 未清理测试数据 | 每个用例后清理 |

### 8.3 元素定位建议

| 优先级 | 定位方式 | 示例 | 稳定性 |
|--------|---------|------|--------|
| 1 | data-testid | `data-testid="login-button"` | ⭐⭐⭐⭐⭐ |
| 2 | ARIA 属性 | `aria-label="登录"` | ⭐⭐⭐⭐ |
| 3 | 文本内容 | "登录按钮" | ⭐⭐⭐ |
| 4 | CSS 类名 | `.btn-primary` | ⭐⭐ |
| 5 | CSS 选择器 | `#login > button` | ⭐ |

**建议**：为关键元素添加 `data-testid` 属性

```tsx
// React 组件示例
<button
  data-testid="login-submit-button"
  onClick={handleSubmit}
>
  登录
</button>
```

```vue
<!-- Vue 组件示例 -->
<button
  data-testid="login-submit-button"
  @click="handleSubmit"
>
  登录
</button>
```

### 11.3 日志查询方法

#### 查看后端日志

```bash

docker compose logs api -f

```

#### 查看 MySQL 日志

```bash

docker compose logs mysql -f

# 进入 MySQL 容器查看

# 查看慢查询日志（如果启用）

```

#### 查看 Redis 日志

```bash
# 查看 Docker 日志
docker compose logs redis -f

# 实时监控 Redis 命令
docker exec -it {redis容器名} redis-cli MONITOR
```
